# Flutter APK size (flutter_app)

**Distribution is direct .apk download** from the VPS (`/uploads/downloads/hrms-latest.apk`, in-app update via version.json). This constrains the size levers — see below.

## What drives size
- **Universal APK**: CI runs `flutter build apk --release` (no ABI split) → native libs (Flutter engine + ML Kit + camera) bundled for arm64-v8a + armeabi-v7a + x86_64 in ONE apk. Single biggest factor.
- **google_mlkit_face_detection** bundles a face model + native libs (~16–20MB across ABIs). Required for face attendance — never remove.
- R8 was off (minify/shrinkResources false) before optimization.

## Safe levers (applied)
- Enable `minifyEnabled true` + `shrinkResources true` in release. Needs ProGuard keeps; **ML Kit/GMS AARs ship their own consumer rules** — do NOT blanket `-keep com.google.android.gms.**` (kills shrink), pin only `com.google.android.gms.internal.mlkit_vision_face.**` + odml + dontwarn.
- Remove packages with zero Dart imports. `GeneratedPluginRegistrant.java` is regenerated by `flutter pub get`, so committed stale copy is overwritten — pubspec is the source of truth.
- `--tree-shake-icons` + `--split-debug-info` in CI: safe, modest.

## Risky levers (NOT auto-applied — explain first)
- **`--split-per-abi` / arm64-only APK**: biggest native-size drop (~50–65%) but drops 32-bit (armeabi-v7a) & x86 devices. Medium risk for direct-download channel.
- **AAB**: cannot be sideloaded/direct-downloaded — only useful if publishing to Google Play. Pointless for this app's direct-.apk distribution.
- **`--obfuscate`**: small extra gain, can break reflection-y code; not worth blind enablement without a test build.

**Why:** Cannot run a Flutter build locally (no SDK) — verification happens only on the next CI build. So broke-functionality risk (face detection, PDF, login) must be checked on a real device after each shrink-config change; keep rollback trivial (flip the two gradle booleans).
