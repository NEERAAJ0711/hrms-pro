## Rule
The Node app runs correctly on port 8080 (internal health check passes), but the web server proxy (nginx or alternative) is NOT configured in `/etc/nginx/`. The CD workflow's nginx fix step never fires.

**Why:** `grep -rl "tbjvisionconnect.com" /etc/nginx/` returns nothing; `find /etc/nginx /home/*/conf -name "*.conf"` also returns nothing. Hostinger VPS may use OpenLiteSpeed, Nginx at a non-standard path, or a panel-managed config.

**How to apply:** The next CD run (#472+) includes `nginx -T` dump, broad `grep -rl` across `/etc /usr/local /home`, process listing, and port scan. Read those logs to identify the actual web server and config location. Then update the workflow to target the right path or add a server-specific config.

**Evidence from run #478 logs:**
- L493: `✅ Server is up on port 8080` — app healthy internally
- L582: `Config file: NOT FOUND` — nginx search found nothing
- L583: `=== ALL NGINX CONFIGS ===` + L584 immediately = find also found nothing
- Domain curl returns 000 (SSL mismatch when hitting raw IP, or 502 via domain)

## Additional findings (run #482, #483)

**ENOSPC root cause:** `npm ci` re-downloads `libtensorflow.so.2.9.1` (344 MB) every deploy.
VPS only had ~858 MB free → ENOSPC mid-extract. The SSH timeout in run #481 was from
`grep -rl` scanning the full filesystem (4+ min → broken pipe).

**nginx location confirmed:** `/home/clp/services/nginx/nginx.conf` (clp-nginx.service).
Domain config search should target that dir only, NOT all of /etc /usr/local /home.

**Fix applied:** rescue `node_modules` from current deploy before backup, restore after
extract, then run `npm install --prefer-offline` instead of `npm ci`. TensorFlow binary
(344 MB) is never re-downloaded after first install.

**Node.js 20 warning** is now just informational — `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true`
in workflow env forces Node.js 24 (message changed from "deprecated" to "being forced to run on").

## Run #484 — ENOSPC fully resolved

- **Root cause of disk shortage**: Previous failed deploys (ENOSPC mid-npm-ci) left large
  partial `node_modules` and backup directories on disk. Cleanup of old backups (via
  `ls -dt hrms-app-backup-* | tail -n +2 | xargs rm -rf`) freed ~17GB (858 MB → 17,724 MB).
- **node_modules rescue pattern**: Move `$APP_DIR/node_modules` to a temp path BEFORE backup,
  restore AFTER extract, then run `npm install --prefer-offline` (not `npm ci`).
  This means tensorflow's 344MB binary is only ever downloaded once.
- **Next blocker**: `/home/clp/services/nginx/` is not accessible to the deploy user (`workeaz...`).
  `grep -rl tbjvisionconnect.com /home/clp/services/nginx` returns nothing — permission denied.
  The CLP panel nginx config must be fixed by root/panel access to set proxy_pass → 127.0.0.1:8080.
- App IS running on port 8080 (PM2 online, health check passes internally).

## FULLY RESOLVED

**Root cause of 502:** `/etc/nginx/sites-enabled/hrms.workeazy.in.conf` had
`proxy_pass http://127.0.0.1:5000/` — pointing to port 5000 (nothing running there).
HRMS runs on port 8080. One `sed` fixed it:
  `sed -i 's|proxy_pass http://127.0.0.1:5000/;|proxy_pass http://127.0.0.1:8080/;|g'`

**hrdrps project** also removed: `hrdrps.conf` deleted, files at
`/home/workeazy-hrms/htdocs/newproject/artifacts/kra-kpi-app/` removed.

**File that controls tbjvisionconnect.com:**
  `/etc/nginx/sites-enabled/hrms.workeazy.in.conf`
  server_name: tbjvisionconnect.com
  proxy_pass: http://127.0.0.1:8080/ (fixed)

**CD pipeline auto-fix note:** The deploy workflow's nginx search targets
`/home/clp/services/nginx/` — this path does NOT exist on this server.
The actual nginx config is in `/etc/nginx/sites-enabled/`. Update the
deploy workflow NGINX_CONF search to also include `/etc/nginx/sites-enabled/`.
